# PVRIG blocker success-case workflow

This directory turns the local success-case mechanism review into an executable
screening workflow for future PVRIG VHH/Fv docking results.

## Purpose

The workflow answers one narrow question:

```text
Does this pose look structurally blocker-like for PVRIG-PVRL2/CD112,
or is it only a PVRIG binder/interface-contacting pose?
```

It does not prove experimental blocking. It produces a success-case-calibrated
screening label that should be followed by 9E6Y re-scoring, leakage checks, and
binding/blocking assays when possible.

## Inputs

Required:

- `--occlusion-csv`: output from `docking/scripts/score_cdr_region_occlusion.py`.

Recommended:

- `--mechanism-csv`: output from `docking/scripts/score_pvrig_vhh_pose.py`, used
  to merge HADDOCK rank/score and hotspot metrics by `model`.

The classifier accepts either current HR-151 column names:

- `total_vhh_pvrl2_residue_pair_occlusion`
- `cdr3_residue_pair_occlusion`
- `cdr3_residue_pair_occlusion_fraction`

or the previous compact names:

- `total_pvrl2_residue_pair_occlusion`
- `cdr3_pvrl2_residue_pair_occlusion`
- `cdr3_occlusion_fraction`

## Run on a new candidate

If you only have a VHH sequence, first prepare a candidate workdir:

```bash
python docking/success_case_validation/prepare_candidate_sequence_workflow.py \
  --name candidate_name \
  --sequence 'HVQL...' \
  --out-root docking/candidates \
  --auto-cdr
```

This writes:

- `inputs/candidate_name_vhh.fasta`
- `inputs/candidate_name_cdr_ranges.csv`
- `inputs/candidate_name_anarci_imgt_H.csv` when `--auto-cdr` is used
- `run_node1_structure_prediction.sh`
- `run_node1_haddock3.sh`
- `haddock3/candidate_name_pvrig_hotspot_test.cfg`
- `haddock3/data/candidate_name_cdr_to_pvrig_hotspot_ambig.tbl`
- `postprocess_after_docking.sh`

`--auto-cdr` uses the local `.conda-envs/ab-data-validator/bin/ANARCI` command
to derive IMGT 27-38, 56-65, and 105-117 CDRs, exact-match them back to the
input FASTA, and write sequence-position ranges. If ANARCI is unavailable, pass
manual ranges with `--cdr1`, `--cdr2`, and `--cdr3`.

`run_node1_structure_prediction.sh` now also normalizes the NanoBodyBuilder2 PDB
to chain A with sequential residue numbers. This is required because the CDR
ranges used by the scorer are sequence-position ranges.
The NanoBodyBuilder2 command uses `-u` to skip a known strained-sidechain
bond-repair path that caused rare ImmuneBuilder/OpenMM failures in mutant
controls; local geometry QC still checks the predicted backbone before docking.

First score a reference baseline. Example for re-aligning 8X6B-aligned poses to
9E6Y using the local hotspot residue map:

```bash
python docking/success_case_validation/score_reference_baseline.py \
  --models cluster_1_model_1,cluster_2_model_1 \
  --pose-dir path/to/top_models_aligned_to_8x6b \
  --pose-pattern '{model}_aligned_to_8x6b.pdb' \
  --output-pose-dir path/to/top_models_aligned_to_9e6y \
  --out-dir path/to/9e6y_baseline_reports \
  --reference-pdb data/structures/9E6Y.pdb \
  --baseline-label 9e6y \
  --mobile-pvrig-chain B \
  --reference-pvrig-chain A \
  --vhh-chain A \
  --reference-pvrl2-chain D \
  --pair-map-csv data/structures/PVRIG_hotspot_set_v1.csv \
  --mobile-ref-column pdb_8x6b_ref \
  --reference-ref-column pdb_9e6y_ref \
  --hotspots-csv data/structures/PVRIG_hotspot_set_v1.csv \
  --hotspot-ref-column pdb_9e6y_ref \
  --rank-score-csv path/to/haddock3_top_model_mechanism_scores.csv
```

Then classify that baseline:

```bash
python docking/success_case_validation/apply_blocker_judgment.py \
  --occlusion-csv path/to/candidate_cdr_region_occlusion_summary.csv \
  --mechanism-csv path/to/candidate_pose_score.csv \
  --candidate-name candidate_name \
  --format-context naked_vhh \
  --out-csv path/to/candidate_blocker_classification.csv \
  --out-md path/to/candidate_blocker_classification.md
```

Use `--format-context` only as an annotation. Valid values are:

```text
naked_vhh, vhh_fc, igg1, igg4, bispecific, unknown
```

## Class labels

- `BLOCKER_LIKE_A`: passes the HR-151-calibrated first-pass VHH docking gate.
- `BLOCKER_PLAUSIBLE_B`: has substantial/partial PVRL2 occlusion but misses at
  least one A-level threshold; inspect and re-score.
- `BINDER_LIKE_C`: hotspot/interface contact exists, but PVRL2 occlusion is too
  low; this catches the HR-151 `cluster_2_model_1` failure mode.
- `FORMAT_CONTEXT_D`: reserved for format-level interpretation; not inferred
  from naked docking alone.
- `EVIDENCE_INFERENCE_ONLY_E`: useful hypothesis only; not enough blocker-like
  geometry in the available columns.

## A-level thresholds

`BLOCKER_LIKE_A` requires all of the following:

- `hotspot_overlap_count >= 14`
- `total_vhh_pvrl2_residue_pair_occlusion >= 500`
- `cdr3_pvrl2_residue_pair_occlusion >= 100`
- `cdr3_occlusion_fraction >= 0.15`

`BINDER_LIKE_C` is assigned when:

```text
hotspot_overlap_count >= 14
and total_vhh_pvrl2_residue_pair_occlusion < 50
```

## Multi-baseline consensus

After generating classification CSVs for both 8X6B and 9E6Y, combine them:

```bash
python docking/success_case_validation/summarize_multibaseline_judgment.py \
  --classification 8x6b=path/to/8x6b_blocker_classification.csv \
  --classification 9e6y=path/to/9e6y_blocker_classification.csv \
  --candidate-name candidate_name \
  --out-csv path/to/candidate_multibaseline_consensus.csv \
  --out-md path/to/candidate_multibaseline_consensus.md
```

Consensus labels:

- `CONSENSUS_BLOCKER_LIKE_A`: at least two baselines support blocker-like geometry.
- `SINGLE_BASELINE_BLOCKER_RECHECK`: one baseline is blocker-like; repeat or inspect
  the other baseline before final ranking.
- `SINGLE_BASELINE_BINDER_LIKE_C`: one baseline is binder-like/nonblocking; repeat
  if needed, but do not prioritize as blocker from this baseline.
- `DISCORDANT_REDOCK_REQUIRED`: one baseline is blocker-like and another is
  binder-like; inspect alignment or redock.
- `DISCORDANT_PLAUSIBLE_VS_BINDER_RECHECK`: one baseline is plausible but
  another is binder-like; do not prioritize without redocking or assay evidence.
- `CONSENSUS_BINDER_LIKE_C`: available baselines support binder-like/nonblocking
  behavior.

## Interpretation boundary

- Passing the screen means structurally blocker-like, not experimentally proven.
- Docking contact residues are inference unless backed by a complex structure or
  epitope map.
- Fc/NK/TIGIT/CD226 effects must be evaluated as separate format/biology layers.
- Final ranking should repeat the same PVRL2 overlay logic against 9E6Y.
